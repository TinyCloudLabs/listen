Single page architecture guide

How the boilerplate turns one wallet approval into authenticated app access.

This worktree is a Bun monorepo for TinyCloud + OpenKey apps. The current branch adds a manifest-driven capability chain: one manifest file defines the app namespace, default capabilities, and backend delegation template. The backend serves a runtime manifest with its DID injected, the frontend signs in once, then issues the backend delegation from the session key. The flow is verified against TinyCloud SDK 2.1.0, where session-key UCAN delegation activation uses the raw UCAN header expected by the TinyCloud host.

Authentication OpenKey connects the wallet, then the backend verifies the SDK-produced SIWE and issues its own session token.
Authorization TinyCloud permissions are declared as manifest entries and converted into recap capabilities for the session.
Delegation The backend serves /api/manifest from the single manifest file, inserting its live DID before sign-in. The resulting UCAN is activated without a Bearer prefix.

Repository shape

The packages are small on purpose.

The boilerplate keeps reusable SDK glue in packages and leaves the concrete app behavior in examples. That split lets a real app replace the model, routes, and UI without rewriting auth, delegation, session storage, or API plumbing.

core

@tinyboilerplate/core

Shared types and constants. It defines the item model, server-info permission shape, delegation status, store type, and default action lists.

client

@tinyboilerplate/client

Browser helpers for OpenKey wallet connection, nonce and SIWE verification, manifest loading/composition, TinyCloud sign-in, delegation submission, session storage, and authenticated fetch.

server

@tinyboilerplate/server

Backend helpers for TinyCloud identity, nonce/session-token auth, CSRF checks, delegation persistence, session refresh, and delegated-access cache.

Manifest capability chain

Sign-in composes permissions before the wallet prompt.

The React app does not sign in first and ask for backend powers later. It fetches the backend runtime manifest, which already contains the backend delegation with the live DID inserted, and passes that manifest into createAndSignIn.

Frontend

What the app contributes

examples/conversation-sync/manifest.json declares the app id, extra app runtime permissions, and a backend delegation template. It omits prefix and defaults, so the SDK uses the normal defaults: prefix equals the manifest id and defaults are enabled.

Backend

What the server advertises

GET /api/manifest returns a runtime manifest with the backend DID inserted into the file's backend delegation template. GET /api/server-info remains a lightweight discovery endpoint for the DID and status.

Key idea: the manifest is the permission planning document. If the backend delegation is inside the capabilities granted to the session key, tcw.delegateTo can produce the backend delegation without asking the wallet again. The UCAN activation request must send the serialized UCAN as the raw Authorization value; adding Bearer makes the host parse a non-UCAN string and reject the activation. Live write-event hooks are declared in the manifest but remain disabled unless VITE_ENABLE_TINYCLOUD_HOOKS=true, because the selected node must advertise and serve the hooks ticket endpoint.

Data paths

The app can use the backend path or direct TinyCloud path.

The backend path is for validation, server-side logic, and shared APIs. The direct path is for browser-owned storage operations that do not need backend mediation.

State diagrams

Runtime state is split between browser session and backend delegation.

Session restore only gives the frontend an API client. It does not recreate a TinyCloudWeb instance, so fresh direct storage or new delegations still require signing in again.

Listen conversation sync

Frontend setup writes secrets through the backend, not into defaults.

Listen runs from examples/conversation-sync. The frontend must be served from https://localhost:5173 for OpenKey/passkeys, while the local backend stays on http://localhost:3001. Fireflies keys are never bundled into the app or defaulted in code; the user enters the key in the setup form, and the backend stores it in that user's TinyCloud KV via delegated access.

Capabilities

Backend permissions are exact

The manifest backend template asks for user KV at / and user SQL database conversations. After manifest resolution those become com.tinycloud.conversation-sync/ and com.tinycloud.conversation-sync/conversations. Conversation routes use conversationSql(access), which calls access.sql.db("com.tinycloud.conversation-sync/conversations") instead of the SDK shortcut database named default.

SDK chain

Backend activation keeps every resource

The browser sends one multi-resource delegation. The node SDK must derive the backend's active session from PortableDelegation.resources, not just the flat first path and actions, or SQL permissions are dropped.

No default secrets

Secrets stay user-scoped

The Fireflies API key is not an environment default and is not in manifest.json. The only persistent copy is the user's TinyCloud KV value at config/fireflies-key, written by the backend after the frontend submits the setup form.

Live hooks

Hooks are host-dependent

The runtime hook grant targets hooks/sql/com.tinycloud.conversation-sync/conversations/conversation. The frontend only mounts the live event stream when the hooks flag is enabled, and it checks POST /hooks/tickets before subscribing so unsupported hosts do not loop on ticket requests.

Build from it

Most app changes are model, manifest, route, and UI changes.

Keep the auth and delegation chain intact, then swap the generic item CRUD pieces for your app-specific resource.

  1. Define the app model in packages/core/src/index.ts. Replace Item and its request/response types with your domain object.
  2. Declare app and backend capabilities in the single manifest. Use app-relative service, space, path, action, and expiry entries. Let the SDK apply the manifest id prefix.
  3. Serve the runtime manifest through /api/manifest. The backend reads the manifest file, injects its DID into the backend delegation template, and returns a normal TinyCloud manifest for sign-in.
  4. Copy and adapt the backend route pattern. Routes run after authMiddleware and delegationMiddleware, then use req.delegatedAccess.
  5. Build the frontend panel with createApiClient or direct tcw calls. Use backend APIs for mediated workflows and direct TinyCloud calls when browser-only storage is enough.
