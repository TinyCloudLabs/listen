Single page architecture guide

How the boilerplate turns one wallet approval into authenticated app access.

This worktree is a Bun monorepo for TinyCloud + OpenKey apps. The current branch adds a manifest-driven capability chain: the frontend composes the app manifest with backend-advertised permissions, signs in once, then issues the backend delegation from the session key. The flow is verified against TinyCloud SDK 2.1.0, where session-key UCAN delegation activation uses the raw UCAN header expected by the TinyCloud host.

Authentication OpenKey connects the wallet, then the backend verifies the SDK-produced SIWE and issues its own session token.
Authorization TinyCloud permissions are declared as manifest entries and converted into recap capabilities for the session.
Delegation The backend advertises what it needs through /api/server-info; the client appends that as a manifest delegation before sign-in and activates the resulting UCAN without a Bearer prefix.

Repository shape

The packages are small on purpose.

The boilerplate keeps reusable SDK glue in packages and leaves the concrete app behavior in examples. That split lets a real app replace the model, routes, and UI without rewriting auth, delegation, session storage, or API plumbing.

core

@tinyboilerplate/core

Shared types and constants. It defines the item model, server-info permission shape, delegation status, store type, and default action lists.

client

@tinyboilerplate/client

Browser helpers for OpenKey wallet connection, nonce and SIWE verification, manifest loading/composition, TinyCloud sign-in, delegation submission, session storage, and authenticated fetch.

server

@tinyboilerplate/server

Backend helpers for TinyCloud identity, nonce/session-token auth, CSRF checks, delegation persistence, session refresh, and delegated-access cache.

Manifest capability chain

Sign-in composes permissions before the wallet prompt.

The React app does not sign in first and ask for backend powers later. It fetches the backend's DID and permission list up front, appends that as a manifest delegation, and passes the composed manifest into createAndSignIn.

Frontend

What the app contributes

The static manifest declares the app's own TinyCloud resources. In this branch, examples/react-express/frontend/public/manifest.json grants KV, SQL, DuckDB, and capability-read access over the default space with an empty path scope.

Backend

What the server advertises

GET /api/server-info returns the backend DID, display name, 7d delegation expiry, and the KV + SQL permissions needed by authenticated data routes.

Key idea: the manifest is the permission planning document. If the backend delegation is inside the capabilities granted to the session key, tcw.delegateTo can produce the backend delegation without asking the wallet again. The UCAN activation request must send the serialized UCAN as the raw Authorization value; adding Bearer makes the host parse a non-UCAN string and reject the activation.

Data paths

The app can use the backend path or direct TinyCloud path.

The backend path is for validation, server-side logic, and shared APIs. The direct path is for browser-owned storage operations that do not need backend mediation.

State diagrams

Runtime state is split between browser session and backend delegation.

Session restore only gives the frontend an API client. It does not recreate a TinyCloudWeb instance, so fresh direct storage or new delegations still require signing in again.

Build from it

Most app changes are model, manifest, route, and UI changes.

Keep the auth and delegation chain intact, then swap the generic item CRUD pieces for your app-specific resource.

  1. Define the app model in packages/core/src/index.ts. Replace Item and its request/response types with your domain object.
  2. Declare app capabilities in the frontend manifest. Use service, space, path, action, and expiry entries that match what the app needs during its session.
  3. Advertise backend needs through /api/server-info. The client composes this response into the manifest before sign-in, so these permissions must match the routes.
  4. Copy and adapt the backend route pattern. Routes run after authMiddleware and delegationMiddleware, then use req.delegatedAccess.
  5. Build the frontend panel with createApiClient or direct tcw calls. Use backend APIs for mediated workflows and direct TinyCloud calls when browser-only storage is enough.
